# Дорожная карта проекта: NIR vs Chaincode

Этот документ разделяет текущие достижения и планы развития на два уровня: общесистемный (NIR) и уровень смарт-контрактов (Chaincode).

---

## 🏛 1. Уровень NIR (Общая ИТ-инфраструктура)
Здесь сосредоточены работы по развертыванию сети, интеграции и внешним интерфейсам.

### ✅ Реализовано
*   **Сеть Fabric**: Развернута многоузловая сеть (Org1, Org2, Orderer).
*   **Контейнеризация**: Полная настройка Docker-окружения для всех компонентов базы.
*   **Скрипты автоматизации**: Готовы инструменты для деплоя (`deploy_chaincode.py`), настройки каналов и крипто-материалов.
*   **Хранилище**: Настроены базы CouchDB с внешним доступом для анализа данных.

### 🚀 Планируется
*   **Web-интерфейс (Frontend)**: Создание полноценного dashboard для визуализации задач.
*   **Интеграция с IPFS**: Создание внешнего хранилища для файлов, ссылки на которые будут в чейнкоде.
*   **Автоматизация CI/CD**: Настройка пайплайнов для автоматического тестирования чейнкода при изменениях.

---

## 📜 2. Уровень Chaincode (Логика смарт-контракта)
Здесь описана "умная" логика, которая выполняется внутри блокчейна.

### ✅ Реализовано (и проверено)
*   **Стандарт SDK**: Полный переход на современный и официально поддерживаемый **TypeScript SDK** (`fabric-contract-api` и `fabric-shim`) вместо сломанных Python-заглушек.
*   **Архитектура CCAAS**: Чейнкод развернут как внешний сервис (Chaincode-As-A-Service), что обеспечивает стабильность и удобство обновления.
*   **CRUD Логика**: Создание, обновление и чтение задач с Ledger с использованием строгой типизации TypeScript.
*   **🛡 Client Identity**: Чейнкод успешно определяет, *кто именно* (MSP ID + X.509 сертификат) вызывает функцию, записывая создателя задачи.
*   **📜 History API**: Реализован и проверен метод получения всей цепочки изменений документа с метками времени.
*   **🔍 Rich Query**: Успешно выполняется поиск по полям (например, "все задачи Петрова") через CouchDB.
*   **📑 Пагинация**: Протестирован механизм закладок (Bookmarks) для запросов, возвращающих большие списки.
*   **🔔 Events**: События (Events) успешно генерируются при создании и обновлении задач, попадая в транзакции.
*   **Детерминированность**: Решена проблема рассинхронизации времени между пирами за счет использования транзакционного времени (`getTxTimestamp`) вместо локального.

### 🚀 Планируемые улучшения (Chaincode)
*   **RBAC (Роли)**: Добавление проверки прав прямо в код (например, "только Admin из Org2 может закрыть задачу").
*   **Консенсус по подписям**: Логика, требующая подтверждения от двух разных участников для смены критического статуса.
*   **State Machine**: Жесткое описание графа статусов (нельзя прыгнуть из "New" сразу в "Finished").
*   **Шифрование данных**: Возможность шифровать часть полей задачи, чтобы их могли видеть только участники с определенными ключами.
