# Руководство по тестированию чейнкода TaskDocument

Данный документ содержит пошаговый план тестирования полного функционала смарт-контракта в сети Hyperledger Fabric. 

Все тесты предполагается выполнять из-под Windows (PowerShell) с использованием обновленного вспомогательного скрипта `invoke_within_peer.sh`.

### 🔄 Базовый цикл тестирования:
Каждый тест выполняется по одному и тому же сценарию:
1. Открываете скрипт `invoke_within_peer.sh` в редакторе.
2. Стираете знак решетки `#` **только** перед нужными строками текущего шага (ненужные должны быть закомментированы).
3. Сохраняете файл (`Ctrl+S`).
4. Выполняете в терминале PowerShell две команды (скопировать скрипт внутрь и запустить его).

**Для шагов, где вы создаете или меняете данные (`invoke_task`):**
```powershell
docker cp ./invoke_within_peer.sh peer0.org1.example.com:/tmp/invoke.sh
docker exec peer0.org1.example.com /tmp/invoke.sh
```

**Для шагов, где вы просто читаете данные (`query_task`), используйте вот эту вторую команду**, чтобы ответ в терминале был красивым и форматированным:
```powershell
docker cp ./invoke_within_peer.sh peer0.org1.example.com:/tmp/invoke.sh
docker exec peer0.org1.example.com /tmp/invoke.sh | ConvertFrom-Json | ConvertTo-Json -Depth 10
```

## 🛡 1. Тестирование Client Identity (Идентификация клиента)
**Цель:** Проверить, что чейнкод четко определяет инициатора.

1. **Создание задачи:** В скрипте раскомментируйте строку под пунктом **`1.1 Создаем`** (начинается с `invoke_task`). Сохраните и запустите в PowerShell.
2. **Проверка: кто создал:** Закомментируйте пункт `1.1` (верните решетку) и раскомментируйте пункт **`1.2 Смотрим поле creator`**. Запустите скрипт. 
**✅ Результат:** В ответе JSON в поле `creator` должен быть записан ваш `Org1MSP` или сертификат.

---

## 📜 2. Тестирование History API (История изменений)
**Цель:** Убедиться, что блокчейн сохраняет абсолютный "след" изменения состояний.

1. **Изменение статусов:** Раскомментируйте сразу **все три строки** под пунктом **`2.1 Создаем и дважды меняем`** (убрав `#` только перед `invoke_task`). Запустите в PowerShell: скрипт за один раз создаст задачу и дважды поменяет ее статус.
2. **Просмотр истории:** Закомментируйте блок `2.1` обратно и раскомментируйте строку **`2.2 Читаем историю`**. Запустите скрипт.
**✅ Результат:** Вернется массив из трех состояний (от NEW до COMPLETED) с уникальными `tx_timestamp`.

---

## 🔍 3. Тестирование Rich Query (Сложные CouchDB-запросы)
**Цель:** Проверить поиск по атрибутам задачи внутри JSON, игнорируя ненужные.

1. **Генерация данных:** Раскомментируйте **все три строки** под пунктом **`3.1 Создаем разные задачи`**. Запустите их через PowerShell.
2. **Выполнение поиска:** Верните комментарии для `3.1` и раскомментируйте строку **`3.2 Ищем ТОЛЬКО задачи Иванова`**. Запустите скрипт.
**✅ Результат:** Выведутся только три задачи для Иванова. Четвёртая (Петрова) будет автоматически отфильтрована на уровне базы.

---

## 📑 4. Тестирование Pagination (Пагинация выборок)
**Цель:** Проверить выдачу данных частями по специальной "закладке" (`bookmark`).

1. **Генерация массива:** Раскомментируйте **все 4 строки** под пунктом **`4.1 Создаем 4 тестовые записи`**. Запустите скрипт.
2. **Первая страница:** Закомментируйте `4.1` и раскомментируйте **`4.2 Получаем ТОЛЬКО первые 2 записи`**. Запустите. Выдаст две задачи. Из самого конца этого ответа вам нужно аккуратно скопировать непонятную строку из поля `bookmark` (обычно начинается с `g1AAAA...`).
3. **Вторая страница:** Закомментируйте `4.2` и раскомментируйте **`4.3 Вставьте ваш bookmark...`**. Прямо в скрипте вместо слова `ВАША_ЗАКЛАДКА` вставьте скопированный код (кавычки вокруг него должны остаться!). Запустите.
**✅ Результат:** Вернется вторая половина задач, начиная с того места, где закончился первый поиск.

---

## 🔔 5. Тестирование Events (Блокчейн-события)
**Цель:** Проверить, что чейнкод не только меняет базу, но и кричит наружу: *"Эй, клиент, задача создана/изменена!"*.

1. В начале файла `invoke_within_peer.sh` (внутри `# Функция для вызова` на 18 строке) нужно добавить один флаг `--waitForEvent`. Команда должна начинаться так:
   ```bash
   peer chaincode invoke -o orderer.example.com:7050 --waitForEvent --ordererTLSHostnameOverride ...
   ```
2. Теперь раскомментируйте внизу скрипта 5.1 команду `invoke_task`, сохраните и запустите 2 команды в PowerShell.
**✅ Результат:** Выполнение будет чуть дольше. По завершении в терминале появятся строки вида `ClientWait -> txid [...] committed with status (VALID)`. Это системное событие, которое доказывает механизм работы Events: терминал "подписался" на уведомления от пира и честно дождался физической записи транзакции в блокчейн. Пользовательские данные самого ивента (`onTaskCreated`) консоль не выводит (их смогут получать ваши Frontend/Backend приложения через официальный Fabric SDK).

## 🚦 6. Тестирование State Machine (Конечный автомат)
**Цель:** Убедиться, что чейнкод не позволит нарушить жизненный цикл задачи.

1. **Генерация данных:** Раскомментируйте пункт **`6.1 Создаем`** в блоке ТЕСТ 6. Запустите скрипт через PowerShell. Будет создана задача со статусом `NEW`.
2. **Нелегальный прыжок:** Закомментируйте 6.1 и раскомментируйте **`6.2 Пытаемся прыгнуть из NEW сразу в CLOSED`**. Запустите.
**✅ Результат:** Транзакция упадет с ошибкой `Invalid status transition from NEW to CLOSED`. Чейнкод защитил нас!
3. **Легальный шаг:** Раскомментируйте **`6.3 Переводим по правилам в IN_PROGRESS`** и выполните скрипт.
**✅ Результат:** Статус успешно изменится.

---

## 🤝 7 & 8. Тестирование Consensus и RBAC (Одобрения и роли)
**Цель:** Проверить, что задачу невозможно закрыть без подписей от обеих организаций, и сделать это может только администратор из Org2.

1. **Подготовка:** Раскомментируйте **`7.1 Переводим задачу в статус REVIEW`**. Запустите скрипт.
2. **Попытка закрытия без консенсуса:** Раскомментируйте **`7.2 Пытаемся закрыть задачу БЕЗ approvals`**. Запустите.
**✅ Результат:** Вызов завершится отказом `Cannot close task. Missing required approvals from both Org1 and Org2`. (либо руганью на RBAC, так как вы из Org1).
3. **Получаем первую подпись (Org1):** Раскомментируйте **`7.3 Добавляем подпись от Org1`** (поскольку наш скрипт сейчас запущен из-под лица Org1). Запустите. 
**✅ Результат:** Успех. Задача одобрена Org1MSP.
4. **Проверка RBAC и подписей Org2:** 
Для имитации участника из Org2, вам потребуется выполнить команды на другом узле (Пире номер 0, организация 2), с другими переменными окружения.
**В НАЧАЛЕ вашего скрипта `invoke_within_peer.sh` (в редакторе) вам нужно:**
- Поменять строку 4 на `export CORE_PEER_LOCALMSPID=Org2MSP`
- Поменять строку 6 на `export CORE_PEER_ADDRESS=peer0.org2.example.com:9051`
Затем раскомментируйте пункт **`7.4 Подпись Org2`** и **`8.1 Закрытие`**. Скопируйте и запустите:
```powershell
docker cp ./invoke_within_peer.sh peer0.org2.example.com:/tmp/invoke.sh
docker exec peer0.org2.example.com /tmp/invoke.sh
```
**✅ Результат:** Цепочка требований выполнена: консенсус получен (`approvals` имеет 2 подписи), вызывающий - из `Org2MSP` (условие RBAC выполнено), следовательно статус задачи успешно изменится на `CLOSED`.
